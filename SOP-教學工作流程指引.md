## 四、懶人包執行 SOP

### 4.1 執行前檢查

每次執行懶人包前，OpenCode 會自動執行 🔁 **重複執行安全檢查**：
- 已安裝的工具 → 自動跳過，不重複安裝
- 未安裝的工具 → 依步驟逐一安裝
- 所有工具已就緒 → 直接跳到最終驗證

### 4.2 通用執行流程

```
步驟零：環境檢查（OS、網路、已安裝工具）
  └──→ 步驟 N：安裝缺失工具（依懶人包不同）
         └──→ 步驟 N+1：功能驗證
                └──→ 完成！接下來你可以這樣用（20 個範例參考）
```

### 4.3 安裝失敗復原

任何懶人包執行失敗時，對 OpenCode 說：

> 「[懶人包名稱] 執行失敗了，幫我檢查哪裡出問題，重新處理。」

OpenCode 會自動：
1. 重新執行環境檢查
2. 找出失敗項目
3. 重新安裝或設定

## 五、版本控制 SOP

### 5.1 開工流程（startup）

對 OpenCode 說 **「開工」**，自動：
1. 確認工作目錄、Git 狀態
2. 讀取 Obsidian 工作筆記（上次進度）
3. 檢查本地與遠端 Git 狀態
4. 產出開工報告

### 5.2 收工流程（shutdown）

對 OpenCode 說 **「收工」**，自動：
1. 整理變更 → 建議 commit 訊息 → 確認後推送
2. 掃描敏感資訊（API Key、token 等）
3. 更新 Obsidian 工作紀錄
4. 產出收工報告

### 5.3 Commit 訊息規範

```
<類型>: <簡要說明>

- <詳細變更 1>
- <詳細變更 2>
```

| 類型 | 使用時機 |
|------|---------|
| `feat` | 新功能、新檔案、新章節 |
| `fix` | 修正錯誤、路徑、日期 |
| `docs` | 文件內容更新、版本提升 |
| `refactor` | 重組內容、重新命名 |
| `chore` | 雜務（.gitignore 等） |

### 5.4 檔案維護原則

| 原則 | 說明 |
|------|------|
| 命名 | `{編號}-{名稱}.md`（不要含 `(1)` 後綴）|
| 編碼 | 繁體中文 UTF-8 |
| Shell | PowerShell 5.1（Windows）、bash（macOS/Linux）|
| 版本 | frontmatter `version` + `date` 每次內容更新時同步 |
| 內部連結 | 建立或重新命名檔案後，同步更新所有 `.md` 內的引用 |
| 狀態 | `初版` → `強化版` → `穩定版`，規劃中檔案標 `v0.0` + `規劃中` |

## 七、常見問題

| 問題 | 解法 |
|------|------|
| 不知道今天要上哪一課 | 查閱教師大綱或 README 懶人包索引 |
| 懶人包檔案在哪裡 | `D:\opencode_data\` 下，依編號命名 |
| 檔案有 ` (1)` 後綴 | 那是舊版命名，已全數移除，如有發現請重新命名 |
| 找不到對應的懶人包 | 查 README.md 的懶人包索引表 |
| 內容需要更新 | 修改對應 .md 檔，更新 version + date，commit 推送 |
| 想新增一個懶人包 | 依 `{編號}-{名稱}.md` 命名，補上 AGENTS.md + README.md 引用 |
| 教案檔案怎麼分工 | 教師大綱（大方向）→ 教師手冊（逐課引導）→ 學生講義（學生用） |
